### Architecture

```
etl_workflow
 │
 ├─ record_pipeline_start()          # wall-clock capture task
 │
 ├─ [parallel fan-out]
 │   ├─ extract_from_database()      ← extract_env (1 CPU/512Mi), Cache(v1, serialize)
 │   ├─ extract_from_api()           ← extract_env (1 CPU/512Mi), Cache(v1, serialize)
 │   └─ extract_from_unknown_source()← extract_env (1 CPU/512Mi), Cache(v1, serialize)
 │                                     returns empty ExtractionResult, never raises
 │
 ├─ merge_extraction_results()       # combines all records, logs source warnings
 │
 ├─ map_task(transform_record)       ← transform_env (2 CPU/1Gi), parallel per-record
 │
 └─ load_to_warehouse()             # writes etl_output.json + returns LoadResult
```

### Key design decisions

| Requirement | Implementation |
|-------------|----------------|
| Two `TaskEnvironment`s | `extract_env = Environment(requests=Resources(cpu="1", mem="512Mi"))` and `transform_env = Environment(requests=Resources(cpu="2", mem="1Gi"))` using `flytekit.core.environment.Environment` (Flyte 2.0 API) |
| Cache extraction tasks | `Cache(version="v1", serialize=True)` passed to each extraction task via the environment; `serialize=True` prevents thundering-herd cache stampedes |
| Unknown source handled gracefully | `extract_from_unknown_source` always returns `ExtractionResult(success=True, records=[])` — the pipeline continues with zero records from that source |
| Parallel transform | `map_task(transform_record)(record=all_records)` fans out one task per record concurrently |
| Source failure recovery | `merge_extraction_results` iterates all three `ExtractionResult` objects; any with `success=False` log a warning and contribute zero records, no exception propagates |
