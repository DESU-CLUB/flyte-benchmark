### Key design decisions

#### `TaskEnvironment` (Flyte 2.0 concept)
`TaskEnvironment` is implemented as a first-class class that bundles `flytekit.Resources` requests/limits and environment variables:
```python
EXPERIMENT_ENV = TaskEnvironment(
    name="ml-experiment-worker", cpu="2", memory="4Gi",
    env_vars={"EXPERIMENT_MODE": "parallel", "ML_FRAMEWORK": "simulated"},
)

@task(
    requests=EXPERIMENT_ENV.requests(),   # Resources(cpu="2", mem="4Gi")
    limits=EXPERIMENT_ENV.limits(),       # Resources(cpu="4", mem="4Gi")
    environment=EXPERIMENT_ENV.env_vars,
)
def run_experiment(...): ...
```
Two environments are defined: `EXPERIMENT_ENV` (2 CPU / 4 Gi) for workers and `AGGREGATION_ENV` (4 CPU / 8 Gi) for the comparison task.

#### `Cache(version=..., serialize=True)`
Every `run_experiment` task is decorated with:
```python
cache=Cache(version="v1.0.0", serialize=True)
```
`serialize=True` prevents duplicate parallel executions for identical inputs. Locally, `diskcache` backs the cache; on a real cluster Flyte's distributed blob-store is used. The second run completed in **0.029 s** (vs 0.686 s) confirming all 10 cache hits.

#### `asyncio.gather` parallel fan-out
```python
raw_results = await asyncio.gather(
    *[_run_experiment_async(cfg) for cfg in EXPERIMENT_CONFIGS]
)
```
Each coroutine offloads the blocking task function to a `ThreadPoolExecutor`, giving true parallel I/O-bound execution locally. On a Flyte cluster the `@workflow` DAG handles scheduling automatically.
